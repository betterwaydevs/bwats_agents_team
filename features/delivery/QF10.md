**What was done**:

- New flow: validates auth, finds one prospect with empty/null `parse_status`, sets it to `"pending"`, returns immediately
- Endpoint now returns in ~0.57s (previously waited for OpenAI parsing which took 5-15s)
- Local Python parser server (`bwats-parser` systemd service) polls every 60s for `parse_status = "pending"` prospects and handles parsing via local Codex
- Verified: endpoint returns correct JSON (`found`, `id`, `status`, `message`)
- Verified: no pending prospects in queue (local parser is keeping up)
- Scheduled task 576 (v1) was already disabled in prior work

**Post-QA fixes (2026-03-06)**:
- **KeyError 'id' fix**: Added `'id' not in data` guard in `xano_client.py:get_pending_prospect` ‚Äî responses where `found` is truthy but `id` is missing now return `None` instead of crashing
- **AC5 retry logic**: Added in-memory attempt counter in `parse_core.py`. On failure, increments count per prospect_id. Under 3 attempts: resets status to `pending` for retry. At 3 attempts: calls `mark_failed` (sets `conflict`) permanently. Counter clears on success or final failure. Resets on service restart (acceptable ‚Äî worst case is 3 more attempts).

## SEC: Security Review
**Date**: 2026-03-06
**Status**: CONDITIONAL APPROVE
**Assigned to**: security-reviewer

### Verdict: CONDITIONAL APPROVE (1 LOW finding)

### Category 1: Authentication & Authorization
**PASS** ‚Äî All 3 Xano parser APIs (`get_for_parsing`, `save_parsed`, `mark_parse_status`) use `N8N_WEBHOOK_TOKEN` via `auth_token` parameter. Token loaded from `.env` at module level, never hardcoded. The Flask HTTP server (`/parse`, `/health`) has NO authentication ‚Äî see Finding F1 below.

### Category 2: Input Validation & Injection
**PASS** ‚Äî `server.py:24` casts `prospect_id` to `int()` which will raise `ValueError` on non-numeric input (Flask returns 500). The HTML cleaner strips `<script>`, `<style>`, comments, and base64 before passing to Codex. `_extract_json` in `codex_runner.py` uses `json.loads()` which is safe against injection. The `subprocess.run` call in `codex_runner.py:105` uses a list (not shell=True), preventing command injection. Temp files use `tempfile.NamedTemporaryFile` (safe).

### Category 3: Secrets & Credentials
**PASS** ‚Äî `.env` is in `.gitignore` and NOT tracked by git. `N8N_WEBHOOK_TOKEN` loaded via `python-dotenv`. No hardcoded secrets in source. Default fallback for `N8N_WEBHOOK_TOKEN` is empty string `''` ‚Äî if unset, API calls will fail auth on Xano side (fail-closed).

### Category 4: Data Exposure
**PASS** ‚Äî Error handling in `parse_core.py:53` catches all exceptions but does NOT log stack traces or expose them to callers. The `mark_failed` method sets status to `'conflict'` without leaking error details. Codex stderr is truncated to 500 chars in error messages (`codex_runner.py:114`) which is reasonable for internal debugging.

### Category 5: Error Handling & Logging
**PASS (with note)** ‚Äî Bare `except Exception` in `parse_core.py:53` silently swallows errors ‚Äî this is intentional for the cron use case (mark as failed and move on). No logging framework is used; failures are tracked via `parse_status = 'conflict'` in Xano. This is acceptable for a cron-based service.

### Category 6: Rate Limiting & DoS
**LOW FINDING (F1)** ‚Äî The Flask server binds `0.0.0.0:8585` with no authentication on `/parse`. Anyone on the network can trigger parsing by hitting `GET /parse`. The cron mode is fine (runs locally). Risk is mitigated by the fact that this runs on a local/Tailscale network, not exposed to the internet. However, an unauthenticated `/parse` endpoint could be used to burn Codex resources.

**F1 ‚Äî Unauthenticated `/parse` endpoint**
- **Severity**: LOW
- **Impact**: Anyone on the local network can trigger prospect parsing, consuming Codex compute time
- **Mitigation (already present)**: Service runs on local/Tailscale network only, not internet-exposed
- **Recommendation**: Add a simple API key check on `/parse` (read from `.env`, compare via header). Not blocking ‚Äî current network isolation is sufficient.

### Category 7: Dependency Security
**PASS** ‚Äî Dependencies are minimal and pinned: `flask==3.0.3`, `requests==2.32.3`, `python-dotenv==1.0.1`, `pytest==8.0.0`. All are well-maintained, widely-used packages. No known critical CVEs at these versions.

### Category 8: XanoScript-specific Patterns
**PASS** ‚Äî The Xano-side changes are minimal: `Add_prospect` now just sets `parse_status = "pending"` and returns. The 3 new Xano APIs use `auth_token` validation server-side. The `function.run parse_unparsed_prospect` call was removed (reducing attack surface by removing the OpenAI call chain from the request path).

### Category 9: Self-Audit
**Confirming scope**: I reviewed all 6 production Python files (`parser.py`, `parse_core.py`, `xano_client.py`, `codex_runner.py`, `html_cleaner.py`, `server.py`), the `.env` structure (redacted), `.gitignore`, `requirements.txt`, and the Xano-side changes described in the DEV notes. The review covers what was actually changed in QF10 ‚Äî the local parser service and the Xano endpoint modification. No files were skipped.
