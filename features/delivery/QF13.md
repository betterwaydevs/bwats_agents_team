# QF13 — Pending Prospect Count Performance — Delivery Report

> **Status**: deployed-v1 | **Date**: 2026-03-03

## Summary

Created a new `prospect_parse_status_counts` GET endpoint that replaces the slow `unparsed_count` endpoint. The new endpoint uses three separate indexed `db.query` COUNT operations instead of a raw SQL query with COALESCE/NULLIF/GROUP BY, enabling index-only scans on the existing btree index on `parse_status`.

### Findings

#### [HIGH]: Unauthenticated state-changing endpoint — 16910_unparsed_and_lock_reset_GET.xs
**File**: `bwats_xano/apis/prospects/16910_unparsed_and_lock_reset_GET.xs`
**Issue**: The `unparsed_and_lock_reset` endpoint has no authentication check. There is no `var.auth_user` verification in the XanoScript. Anyone who discovers the endpoint URL can reset ALL parsing/conflict prospects to pending status.
**Impact**: An unauthenticated caller can disrupt the parsing pipeline by resetting all in-progress work back to pending, causing re-processing overhead and potential data inconsistency.
**Recommendation**: Add Xano authentication precondition (require valid auth token → `var.auth_user`). Update the frontend `resetParsingQueue()` function in `parsingApi.ts` to pass the auth token (matching the pattern in `getParsingStats()`).
**Status**: SHOULD FIX — The operation is non-destructive (only resets status to "pending"), but production endpoints should require authentication as a principle. Acceptable to ship to QA now, but must be addressed before broader rollout.

#### [MEDIUM]: Frontend reset call sends no authentication — parsingApi.ts:37-46
**File**: `nearshore-talent-compass/src/services/parsingApi.ts:37-46`
**Issue**: The `resetParsingQueue()` function makes a bare `fetch()` call with no `Authorization` header, unlike `getParsingStats()` which properly includes the token. This is inconsistent and will break once the backend adds auth.
**Impact**: Currently functional because the backend also lacks auth. Once backend auth is added, this will fail silently or return 401.
**Recommendation**: Accept `token: string` parameter (like `getParsingStats`) and include `Authorization` header. Wire through from the React component via `useAuth()`.
**Status**: SHOULD FIX (together with the backend auth fix above)

#### [LOW]: GET method used for state-changing operation — 16910_unparsed_and_lock_reset_GET.xs
**File**: `bwats_xano/apis/prospects/16910_unparsed_and_lock_reset_GET.xs:2`
**Issue**: The reset endpoint uses HTTP GET for an operation that modifies database state. GET requests should be idempotent and safe. This makes the endpoint susceptible to accidental invocation via browser prefetch, link preview crawlers, or CSRF.
**Impact**: Low — the endpoint is internal-use only and the reset operation is non-destructive. But it violates HTTP semantics.
**Recommendation**: Consider changing to POST in a future iteration.
**Status**: OPTIONAL
