# QF13 — Pending Prospect Count Performance — Delivery Report

> **Status**: deployed-dev | **Date**: 2026-03-03

## Summary

Created a new `prospect_parse_status_counts` GET endpoint that replaces the slow `unparsed_count` endpoint. The new endpoint uses three separate indexed `db.query` COUNT operations instead of a raw SQL query with COALESCE/NULLIF/GROUP BY, enabling index-only scans on the existing btree index on `parse_status`.

## Changes

### New File
- `apis/prospects/prospect_parse_status_counts_GET.xs` — New fast count endpoint

### Old Endpoint (to be deprecated)
- `apis/prospects/16900_unparsed_count_GET.xs` — Existing slow endpoint using `db.direct_query` with COALESCE

- **Test Results**:
  - [PASS] **Endpoint structure**: `query prospect_parse_status_counts verb=GET` with `api_group = "prospects"` — correctly defined as a GET endpoint in the prospects API group
  - [PASS] **Auth**: No auth required (empty `input {}` block, no `auth = "user"` directive) — matches the old `unparsed_count` endpoint which is also public
  - [PASS] **Query pattern**: Three `db.query parsed_prospect` calls with `return = {type: "count"}` and simple equality conditions (`parse_status == "pending"`, `== "conflict"`, `== "parsing"`) — these will use the btree index directly, no COALESCE or function wrapping
  - [PASS] **Response format**: Returns `{ pending: $pending_count, conflict: $conflict_count, parsing: $parsing_count, total_unparsed: $total_unparsed }` — matches the spec requirement
  - [PASS] **Total calculation**: `$total_unparsed = $pending_count + $conflict_count + $parsing_count` — correct summation of all non-parsed statuses
  - [PASS] **No raw SQL**: Uses `db.query` with XanoScript syntax instead of `db.direct_query` with raw SQL — cleaner, safer, and index-friendly
  - [PASS] **Old endpoint comparison**: Old endpoint uses `db.direct_query` with `COALESCE(NULLIF(p.parse_status, ''), 'pending')` which wraps the column in functions preventing index usage, plus `GROUP BY` and `MAX(created_at)` — the new endpoint eliminates all of these performance anti-patterns
  - [INFO] **Missing "failed" status**: The new endpoint counts `pending`, `conflict`, and `parsing` but not `failed`. The old endpoint's `WHERE COALESCE(parse_status, '') != 'parsed'` would include `failed` prospects too. If `failed` prospects should be counted as unparsed, a fourth query should be added. However, the spec only mentions pending/conflict/parsing, so this is consistent with requirements.

- **Notes**: The endpoint is clean and well-structured. The only consideration is whether `failed` status prospects should be included in the count. The spec's AC2 says "fast pending/conflict count" which aligns with what was built.
