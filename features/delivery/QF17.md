### Findings

#### [LOW]: Pre-existing hardcoded auth token in webhook URL — UnparsedProspectManager.js:13
**File**: `bw_cold_recruiting/classes/UnparsedProspectManager.js:13`
**Issue**: The `parseWebhookUrl` contains a hardcoded `auth_token` query parameter: `?auth_token=AYfIMbL7...`. This token is also present in `AutoCaptureManager.js:421`, `popup.html:778`, and `tenant_configs/bwats.json:7`. **This is NOT a new finding** — it predates QF17 and was not introduced by this change.
**Impact**: If the extension source code is inspected (e.g., from a `.crx` file), the token is exposed. Since this is a Chrome extension shipped as a package, the token is effectively public.
**Recommendation**: Move webhook auth to backend-side validation (e.g., require user auth token instead of a static webhook token). Track as a separate security hardening task.
**Status**: CONSIDER FIXING (pre-existing, not blocking QF17)
