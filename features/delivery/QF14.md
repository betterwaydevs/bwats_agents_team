# QF14 — Add_prospect db.patch Type Error Fix — Delivery Report

> **Status**: dev-deployed | **Date**: 2026-03-03

## Summary

Fixed the `parse_unparsed_prospect` function (ID 11189) to handle AI parser output where array-typed fields (`skills`, `work_history`, `education`, `certifications`, `languages`, `industries`) arrive as JSON strings instead of parsed arrays. This eliminates the `db.patch` type error that was causing prospects to be incorrectly marked as `conflict` status.

## Changes

### Modified File
- `functions/11189_parse_unparsed_prospect.xs` — Added two-layer defensive parsing for array fields

### Change Details
1. **Early-stage parsing (line ~225)**: Pre-parses all 6 array fields on the `parsed` object before date/number normalization runs, so downstream processing still applies correctly
2. **Safety net before return (line ~309)**: Second defense layer on `dbData` right before `return dbData`, catching any edge case where a string might slip through after the copy

## Sign-offs

### DEV Sign-off
- **Agent**: backend-developer
- **Date**: 2026-03-02
- **Status**: PASS
- **Notes**: Root cause identified — the AI parser (`call_open_ia`) occasionally returns array fields as JSON strings (e.g., `"[{\"name\":\"Python\"}]"` instead of `[{"name":"Python"}]`). When `db.patch` receives a string for a column expecting an array, Xano's PHP layer throws a type error. The fix adds defensive JSON parsing at two points in the `api.lambda` that builds `$db_update_data`. The fix also handles edge cases: plain objects (not arrays) get wrapped in `[obj]`, unparseable strings default to `[]`.
