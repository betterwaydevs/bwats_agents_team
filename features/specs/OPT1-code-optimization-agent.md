# OPT1: Code Optimization Agent — Post-DEV Performance & Quality Gate

**Status**: Draft
**Priority**: Medium
**Owner**: TBD
**Created**: 2026-03-03
**Updated**: 2026-03-03

## Overview

Add a code optimization agent to the delivery workflow that reviews and optimizes code generated by developers before it reaches QA. This agent ensures code quality, performance, and adherence to best practices.

## Problem Statement

Currently, the delivery pipeline flows directly from DEV → QA → PO. There's no automated step to verify that generated code is optimized for:
- Performance (algorithm efficiency, resource usage)
- Code quality (clean code principles, maintainability)
- Best practices (language-specific patterns, security)
- Technical debt prevention

This can result in code that works but isn't optimal, requiring later refactoring or causing performance issues.

## Goals

1. Add an optimization agent between DEV and QA stages
2. Automate code quality and performance checks
3. Provide actionable optimization suggestions to developers
4. Ensure optimized code reaches QA (reducing QA cycles)
5. Maintain delivery velocity while improving code quality

## Proposed Solution

### New Pipeline Flow
```
PM → DEV → **OPTIMIZER** → QA → PO → User
```

### Agent Responsibilities
The optimization agent will:
- Review code changes from the DEV stage
- Analyze for:
  - Performance bottlenecks
  - Code smells and anti-patterns
  - Security vulnerabilities
  - Best practice violations
  - Unnecessary complexity
- Provide optimization recommendations
- Either:
  - **Approve** → forward to QA
  - **Request optimization** → send back to DEV with suggestions

### Decision: Auto-fix vs. Review-only
**TBD**: Should the optimizer:
- A) Auto-apply optimizations and notify DEV

## Acceptance Criteria

### AC1: Pipeline Integration
- [ ] Optimization agent is invoked automatically after DEV sign-off
- [ ] Pipeline does not proceed to QA until optimizer approves
- [ ] Delivery logs include optimizer stage with sign-off

### AC2: Optimization Analysis
- [ ] Agent analyzes code changes (files modified in the task)
- [ ] Agent identifies optimization opportunities in categories:
  - Performance
  - Code quality
  - Security
  - Best practices
- [ ] Agent provides specific, actionable recommendations with code examples

### AC3: Gate Enforcement
- [ ] If optimizations are recommended, code returns to DEV stage
- [ ] DEV receives clear list of suggested optimizations
- [ ] DEV can review, apply, and re-submit for optimization review
- [ ] If code is already optimized, agent approves and forwards to QA

### AC4: Reporting
- [ ] Optimizer generates a report with:
  - Date/time of review
  - Files analyzed
  - Optimizations found (or "no optimizations needed")
  - Recommendations (if any)
  - Final decision (approved / request changes)
- [ ] Report is stored in delivery log

## Technical Considerations

### Scope of Analysis
- **In scope**: Code changed in the current task (git diff against base branch)
- **Out of scope**: Full codebase audit (too expensive)

### Analysis Techniques
- Static code analysis
- Pattern matching for common anti-patterns
- Algorithm complexity analysis
- Framework-specific best practices (React, XanoScript, etc.)

### Integration Points
- **Input**: DEV stage sign-off + git branch/commit
- **Output**: Optimization report + approve/reject decision
- **Communication**: SendMessage to DEV if changes requested

### Agent Context
The optimizer agent needs access to:
- Changed files (git diff)
- Project-specific conventions (CLAUDE.md files)
- Language/framework documentation
- Performance baselines (if available)

## Dependencies

- W1 (Multi-Agent Team Delivery Workflow) — must be complete first
- Delivery log format update to include optimizer stage

## Non-Goals (for v1)

- Runtime performance profiling (only static analysis)
- Automated refactoring of unchanged code
- Cross-project optimization (each project analyzed independently)
- Performance benchmarking (just code review)

## Open Questions

1. **Auto-fix vs. suggestions**: Should the optimizer auto-apply safe optimizations, or always require DEV approval?
2. **Failure threshold**: How many/what severity of issues should block QA? Or is it always advisory?
3. **Model choice**: Sonnet for quality, or Haiku for speed? Cost vs. thoroughness trade-off.
4. **Exemptions**: Should certain tasks (e.g., prototypes, experiments) skip optimization?
5. **Scope creep prevention**: How do we ensure the optimizer doesn't over-optimize or suggest unnecessary refactors?
