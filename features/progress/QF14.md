# QF14 — Add_prospect db.patch Type Error — Progress

> **Status**: Fix written locally, pending deployment to Xano
> **Date**: 2026-03-02

## Investigation

### Root Cause

The `parse_unparsed_prospect` function (ID 11189) contains an `api.lambda` that processes AI output and builds a `$db_update_data` object. This object is then passed to:

```xanoscript
db.patch parsed_prospect {
  field_name = "id"
  field_value = $prospect.id
  data = $db_update_data
}
```

The six array-typed database columns are: `skills`, `work_history`, `education`, `certifications`, `languages`, `industries`.

The AI parser (`call_open_ia`) occasionally returns one or more of these fields as **JSON strings** instead of parsed arrays. For example:

- Expected: `[{"name": "Python", "months_experience": 36}]`
- Received: `"[{\"name\": \"Python\", \"months_experience\": 36}]"`

The lambda that builds `dbData` copies these fields as-is from the AI output via a `validFields.forEach` loop. When `db.patch` receives a string for a column that expects an array type, Xano's PHP layer throws:

```
xano\helper\DBO::set(): Argument #2 ($b) must be of type array, string given, called in Patch.php on line 99
```

This error is caught by the `try_catch` block, and the prospect gets marked as `parse_status = "conflict"`.

### Which Field(s)

Any of the six array fields could trigger this, depending on how the AI formats its response for a given prospect. The error message does not specify which field — it just says "string given" where "array" was expected. Prospect 72100 triggers it, but the exact field would require inspecting its raw AI output. The fix handles ALL six fields defensively.

## Fix Applied

### File Modified

`/home/pablo/projects/bwats/bwats_xano/functions/11189_parse_unparsed_prospect.xs`

### Change 1: Early-stage parsing (line ~225)

Added a pre-parse block **before** the existing `work_history` and `skills` normalization code. This ensures that if the AI returns array fields as JSON strings, they get parsed into real arrays before date/number normalization runs.

```javascript
// QF14: Pre-parse any array fields that arrive as JSON strings
// so that downstream normalization (dates, numbers) still applies
["skills", "work_history", "education", "certifications", "languages", "industries"].forEach(function(f) {
  if (parsed[f] !== undefined && parsed[f] !== null && typeof parsed[f] === "string") {
    var t = parsed[f].trim();
    if (t.startsWith("[") || t.startsWith("{")) {
      try { parsed[f] = JSON.parse(t); } catch(e) { parsed[f] = []; }
    } else { parsed[f] = []; }
  }
  if (parsed[f] !== null && parsed[f] !== undefined && typeof parsed[f] === "object" && !Array.isArray(parsed[f])) {
    parsed[f] = [parsed[f]];
  }
});
```

### Change 2: Safety net before return (line ~309)

Added a second defense layer on `dbData` right before `return dbData;`. This catches any edge case where a string might slip through after the copy to `dbData`.

```javascript
// QF14 fix: Ensure array fields are actual arrays, not JSON strings.
var arrayFields = ["skills", "work_history", "education", "certifications", "languages", "industries"];
arrayFields.forEach(function(f) {
  if (dbData[f] !== undefined && dbData[f] !== null) {
    if (typeof dbData[f] === "string") {
      var trimmed = dbData[f].trim();
      if (trimmed.startsWith("[") || trimmed.startsWith("{")) {
        try {
          dbData[f] = JSON.parse(trimmed);
        } catch(e) {
          dbData[f] = [];
        }
      } else {
        dbData[f] = [];
      }
    }
    // Ensure it's an array (not a plain object) — wrap objects in an array
    if (dbData[f] !== null && typeof dbData[f] === "object" && !Array.isArray(dbData[f])) {
      dbData[f] = [dbData[f]];
    }
  }
});
```

### Logic Summary

For each of the 6 array fields, the fix:
1. Checks if the value is a string
2. If yes, attempts `JSON.parse()` — if that fails, defaults to empty array `[]`
3. If the value is a plain object (not an array), wraps it in an array `[obj]`
4. If already an array, leaves it unchanged

This runs in two places:
- **Early** (on `parsed`): so date/number normalization still applies to the parsed data
- **Late** (on `dbData`): as a safety net before passing to `db.patch`
