## Investigation: Current Slow Approach

### Which endpoint does the extension call?

The Chrome extension (`bw_cold_recruiting`) calls:
```
GET https://xano.atlanticsoft.co/api:zE_czJ22/unparsed_count
```

This is defined in:
- **Extension class**: `/home/pablo/projects/bwats/bw_cold_recruiting/classes/UnparsedProspectManager.js` (line 12)
- **Xano endpoint**: `/home/pablo/projects/bwats/bwats_xano/apis/prospects/16900_unparsed_count_GET.xs`

### Why is it slow?

The existing endpoint uses a raw SQL query via `db.direct_query`:

```sql
SELECT
  COALESCE(NULLIF(p.parse_status, ''), 'pending') AS status_group,
  COUNT(*) AS prospect_count,
  MAX(p.created_at) AS newest_prospect_date
FROM x6_146 AS p
WHERE COALESCE(p.parse_status, '') != 'parsed'
GROUP BY 1
ORDER BY 1;
```

**Performance problems:**
1. **COALESCE/NULLIF prevents index usage**: The `COALESCE(NULLIF(p.parse_status, ''), 'pending')` expression wraps the column in functions, which prevents PostgreSQL from using the existing btree index on `parse_status`. The query planner must do a sequential scan instead.
2. **GROUP BY across entire table**: Groups by the computed expression, requiring a full table scan and hash/sort aggregation.
3. **MAX(created_at) adds overhead**: Computing the maximum `created_at` per status group requires examining every matching row's timestamp.
4. **No authentication**: The endpoint is public, which is fine for reads but means anyone can hit it.

### Existing infrastructure (good news)

The `parsed_prospect` table (`x6_146`) already has a **btree index on `parse_status`** (defined in table schema line 153). The `parse_status` enum has values: `pending`, `parsed`, `failed`, `parsing`, `conflict`. This means a simple equality query like `WHERE parse_status = 'pending'` WILL use the index efficiently.

## New Endpoint: `prospect_parse_status_counts`

### Design

- **Method**: GET
- **API Group**: prospects (canonical: `zE_czJ22`)
- **Auth**: None (public, same as current endpoint)
- **File**: `/home/pablo/projects/bwats/bwats_xano/apis/prospects/prospect_parse_status_counts_GET.xs`

### How it works

Instead of one raw SQL query with COALESCE and GROUP BY, the new endpoint runs three separate `db.query` calls with `return = {type: "count"}`:

1. `db.query parsed_prospect WHERE parse_status == "pending"` -> `$pending_count`
2. `db.query parsed_prospect WHERE parse_status == "conflict"` -> `$conflict_count`
3. `db.query parsed_prospect WHERE parse_status == "parsing"` -> `$parsing_count`

Each of these directly hits the btree index on `parse_status` with a simple equality match. No function wrapping, no GROUP BY, no MAX aggregation.

### Response format

```json
{
  "pending": 42,
  "conflict": 3,
  "parsing": 1,
  "total_unparsed": 46
}
```

### Why this is faster

- **Direct index scan**: Each `db.query` with `type: "count"` translates to `SELECT COUNT(*) FROM parsed_prospect WHERE parse_status = 'X'`, which PostgreSQL executes as an index-only scan on the btree index.
- **No function wrapping**: No COALESCE/NULLIF means the query planner can use the index.
- **No MAX**: Removed the `MAX(created_at)` computation entirely since it's not needed for the count display.
- **Three fast queries vs one slow query**: Three index-only count scans are far faster than one sequential scan with aggregation.

**Update the `getUnparsedCount()` method** (lines 61-123) to handle the new flat response format:

The current method expects an array of `{ status_group, prospect_count, newest_prospect_date }` objects. The new endpoint returns a flat object `{ pending, conflict, parsing, total_unparsed }`. The method needs to be updated to:
1. Parse the flat response into the `statuses` array format the popup expects, OR
2. Return the flat counts directly and update `popup.js` to render them differently.

**Recommended approach**: Update `getUnparsedCount()` to convert the new response format into the existing `statuses` array format for backward compatibility with the popup rendering:

```javascript
async getUnparsedCount() {
    try {
        const response = await fetch(this.countApiUrl, {
            method: 'GET',
            headers: { 'Content-Type': 'application/json' }
        });
        if (!response.ok) {
            return { count: 0, newestDate: null, success: false, statuses: [], error: `API request failed: ${response.status}` };
        }
        const data = await response.json();
        // Convert flat response to statuses array for popup compatibility
        const statuses = [];
        if (data.pending > 0) statuses.push({ statusGroup: 'pending', count: data.pending, newestDate: null });
        if (data.conflict > 0) statuses.push({ statusGroup: 'conflict', count: data.conflict, newestDate: null });
        if (data.parsing > 0) statuses.push({ statusGroup: 'parsing', count: data.parsing, newestDate: null });
        return {
            count: data.total_unparsed || 0,
            newestDate: null,
            success: true,
            statuses
        };
    } catch (error) {
        return { count: 0, newestDate: null, success: false, statuses: [], error: error.message };
    }
}
```

**Note**: The `newestDate` field is dropped in the new endpoint for performance. The extension popup currently shows "Last Update" per status, but this will show "No date available" with the new endpoint. This is an acceptable tradeoff for performance.

**Version bump**: The extension `manifest.json` version must be incremented when making this change.
