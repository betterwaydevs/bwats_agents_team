PASS AC1: Second email to same person uses the same sending account as the first

Logic Trace

When a pending email is dequeued, the new code executes the following sticky-account lookup before any round-robin logic:

  1. Query email_send_log (lines 116-121): Queries email_send_log for the most recent record where recipient == $email.to_email and status == "sent", sorted by sent_at desc, returning a single record with email_account_id.
  2. Validate sticky account (lines 124-145): A lambda checks whether the found account:
    • Exists in the active $all_accounts list (implicitly confirming active == true since the list was filtered at line 39)
    • Matches the required account_type of the email
    • Is under its daily_limit (cap check: daily_sent_count < daily_limit)
    If all checks pass, returns the sticky account ID. Otherwise returns null.
  3. Assignment (lines 148-149): $selected_account_id is set to $sticky_account_id. If non-null, the conditional on line 152 is skipped entirely, and the email is sent using the sticky account.

PASS AC2: Falls back to round-robin only if sticky account is inactive or over limit

Logic Trace

The validation lambda (lines 124-145) returns null in four distinct fallback scenarios:

  1. No prior send found — $sticky_log is null or has no email_account_id (first-time recipient)
  2. Account deactivated — the sticky account ID is not found in $all_accounts (which only contains active == true accounts, filtered at line 39)
  3. Account type mismatch — the sticky account's account_type does not match the email's required type
  4. Over daily cap — daily_sent_count >= daily_limit

When $sticky_account_id is null, $selected_account_id is set to null (line 148-149), and the conditional block on line 152 fires, entering the round-robin fallback path (lines 154-194).
