Root Cause Analysis

Stickiness Logic Exists But Is Not Taking Effect

The process_email.xs function (lines 116-145) contains the correct stickiness logic:

  1. Query email_send_log for most recent "sent" to same recipient
  2. Validate the found account is active, correct type, and under daily cap
  3. Use it if available, otherwise fall back to round-robin

Possible causes:

  • The email_send_log query may not be finding records — perhaps the recipient field is stored differently than the to_email value (e.g., casing, trimming, or the field name changed in the schema).
  • The sticky account validation lambda may be returning null — the account_type comparison could be failing if the queued email's account_type is null or different from what is stored on the account.
  • The $all_accounts variable may not match by ID — the accounts.find(a => a.id === stickyId) uses strict equality; if one is a number and the other a string, it would fail silently.
  • The Xano development branch may not have the latest code deployed — the XanoScript file exists locally but the live development environment might be running an older version without the stickiness patch.

Recommended next step: Add debug logging inside process_email.xs after the sticky_log query (line 121) and the sticky_account_id lambda (line 145) to confirm what values are being returned. Also verify the development branch has the latest code pushed via Xano MCP.
