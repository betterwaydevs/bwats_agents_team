BUG Report

BUG-5: API Response Format Mismatch (CRITICAL for AC5/AC6)

Severity: HIGH — Blocks tool card rendering

Location: Backend API GET /api:3Bq6OWvc:development/tools/list

Root Cause: The API returns a flat array [{...}, {...}, {...}], but the frontend service (downloadableToolsApi.ts) expects the response wrapped in an object: {"tools": [{...}, {...}, {...}]}

Frontend code (line 83 of BwatsDownloads.tsx):

const tools = data?.tools;  // data IS the array, so data.tools is undefined
// ...
tools && tools.length > 0 ? /* render cards */ : /* show empty state */

Evidence:

  • API curl returns: [{"id":1,"name":"Linked Communication",...}, {"id":2,...}, {"id":3,...}] (flat array, 3 items)
  • Frontend shows: "No downloads available" (empty state)
  • data?.tools evaluates to undefined because data is the array itself, not a wrapper object

Fix options:

  1. Backend fix (recommended): Wrap API response in {"tools": [...]} to match the spec and the ToolsListResponse TypeScript interface
  2. Frontend fix: Change const tools = data?.tools to const tools = Array.isArray(data) ? data : data?.tools

Design Notes

Upload Endpoint Limitation

The upload endpoint (POST /tools/upload) accepts Xano attachment objects (JSON with {path, name, size, mime, type, meta}) rather than raw multipart file uploads. This means:

  • The file must already exist in Xano's vault/file storage
  • The endpoint updates the tool record's metadata (version, download_url, file_size) but does not handle actual file storage
  • To truly upload a new file, the file must first be placed in Xano's vault through the Xano UI or file management API, then the tools/upload endpoint is called to update the reference

This works for the current use case (admin-only uploads via Claude Code or Xano UI) but differs from the spec's multipart upload description.

Download Endpoint Behavior

The download endpoint returns JSON metadata (the full tool record) rather than a binary file stream with Content-Disposition header as described in the spec. The frontend uses window.open(tool.download_url) from the list response, so the download endpoint is used as a "get tool info by slug" endpoint rather than a file server.
