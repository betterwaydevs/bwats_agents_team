Bug Description
URL: /profile/juliano-tech-lead--Hnxjc5wB1gtBcfFRFiOy?source=candidates
Issue 1: When ?source=candidates was explicitly set in the URL, the fallback to search the prospects index was skipped due to a !sourceParam guard in the conditional.
Issue 2: Even after fixing the fallback, getFullParsedProspect() threw an unhandled error (HTTP 404) because the prospect existed in ElasticSearch but not in the Xano parsed_prospect table. The candidate branch had a try/catch but the prospect branch did not.
Impact: Any profile accessed via a stale or cross-index link showed "Error Loading Profile" instead of the person's data.
Fix Applied (2 changes in BwatsProfile.tsx)
Fix 1: Removed && !sourceParam guard — fallback now always tries the other index regardless of whether ?source= is in the URL.
Fix 2: Wrapped getFullParsedProspect() in try/catch (matching the candidate branch) so the profile still renders from ES data when the Xano record is missing.
