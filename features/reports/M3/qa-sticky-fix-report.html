Context

Function: process_email (ID: 10568, file: functions/communications/10568_process_email.xs)

What changed: The account_type matching gate was removed from the sticky email lambda (lines 124-142). Previously, stickiness only applied when the queued email's account_type matched the sticky account's type. Now the most recent email account used for a recipient becomes sticky for ALL future emails, regardless of account type.

Why: When a candidate applies to a job, they receive an auto-reply from application_thank_you account (ID 32, jobs@email.betterway.dev). Later outreach emails using outbound_general type should reuse the same sender so the recipient sees a consistent "From" address.

Sticky Lambda (Current Code, Lines 124-142)

// Check if sticky account is active and under its daily limit
api.lambda {
  code = """
      const stickyLog = $var.sticky_log;
      if (!stickyLog || !stickyLog.email_account_id) return null;

      const stickyId = stickyLog.email_account_id;
      const accounts = $var.all_accounts || [];

      const match = accounts.find(a => a.id === stickyId);
      if (!match) return null;

      // No account_type check -- sticky account is reused regardless of type
      const underCap = (match.daily_sent_count || 0) < (match.daily_limit || 50);
      if (!underCap) return null;

      return stickyId;
    """
  timeout = 10
} as $sticky_account_id
